# ADR-2026-03-27: Project State Layer for Core

## Status

Accepted

## Date

2026-03-27

## Context

The runtime-governance slice is now in a stable state.

Current verified repo baseline:

- commit: `d2dade3`
- branch: `main`

The closed slice is:

- runtime governance
- manifest layer
- registry layer
- registry-aware documentation sync
- smoke-verified sync/governance flows

Even with that progress, the project still has one repeated operational problem:

- current state is recoverable from git, docs, artifacts, TODO, LOG, and chat
- but there is no single compact operational state file that answers:
  - where we are now
  - what was finished
  - what the active slice is
  - what the next step is
  - what the last green checks were
  - which files are the source of truth

This causes friction when:

- moving to a new chat branch
- resuming after a pause
- checking the current active slice
- deciding what to do next
- avoiding repeated manual checkpoint summaries
- distinguishing active truth from older artifacts

Git history is necessary, but not sufficient for this role.

Artifacts are necessary, but too granular.

Chat summaries are useful, but not a durable project control plane.

The project needs a compact, explicit, refreshable state layer.

---

## Decision

We will add a small **project-state layer** for the `core` topic.

This layer will **not** replace:

- git history
- ADRs
- TODO
- LOG
- memory files
- artifacts

It will sit **above** them as a compact operational snapshot.

The project-state layer will have two outputs:

1. a machine-readable current snapshot
2. a short human-readable current snapshot

This layer will start as a manual/explicit refresh mechanism, not a fully automatic background system.

---

## State Model

### Human-readable state

The existing human context remains important:

- `core/memory/context.md`
- `core/TODO.md`
- `core/LOG.md`

These remain the narrative and working-memory layer.

### Machine-readable state

The new compact state layer will live under:

- `core/artifacts/project-state/latest.json`
- `core/artifacts/project-state/latest.md`

Optional historical snapshots may also be stored under timestamped paths such as:

- `core/artifacts/project-state/<timestamp>/state.json`
- `core/artifacts/project-state/<timestamp>/state.md`

`latest.*` is the current operational truth for quick orientation.

Timestamped copies are historical breadcrumbs, not the main entry point.

---

## Ownership and Refresh Rule

The state snapshot will be generated by a dedicated helper:

- `/home/agent/bin/project-state-refresh`

That helper will refresh the current project state for the `core` topic.

The refresh is expected after any significant milestone, especially after:

- a completed architectural slice
- a green regression or smoke verification
- a commit/push that closes a checkpoint
- a change in current focus
- a change in next-step direction

This means project state becomes a maintained operational layer rather than an occasional manual note.

---

## Minimum Contract for `latest.json`

The first implementation of `latest.json` should contain at least:

- `schema`
- `generated_at_utc`
- `topic`
- `repo_root`
- `git_branch`
- `git_head`
- `git_clean`
- `current_focus`
- `current_slice`
- `last_completed_step`
- `next_step`
- `last_green_checks`
- `key_paths`
- `open_risks`
- `notes`

It may later expand, but the first version should stay compact and stable.

---

## Minimum Contract for `latest.md`

The human-readable state snapshot should answer, in short form:

- where we are now
- what was completed
- what is currently active
- what the next step is
- what was the last verified green state
- what files/paths are the source of truth

It should be compact enough to paste into a new working branch when needed.

---

## Source of Truth Rules

The project-state layer must follow these rules:

1. It summarizes project state.
2. It does not replace git.
3. It does not replace ADRs.
4. It does not replace detailed artifacts.
5. It must point to source-of-truth files and paths.
6. It must stay compact.
7. It must not become a dump of raw logs.

The hierarchy is:

- code and files are the implementation truth
- ADRs are decision truth
- TODO and LOG are working-process truth
- project-state is compact operational orientation truth

---

## Safety Rules

The project-state layer must not store:

- secrets
- tokens
- passwords
- raw log dumps
- very large command output
- temporary diagnostic noise

It should contain only curated, compact state.

---

## Immediate Scope

The first scope is intentionally small.

We are not building a broad orchestration system here.

We are building:

- one stable contract
- one refresh helper
- one current machine-readable snapshot
- one current human-readable snapshot

This is enough to stop repeated manual checkpoint rebuilding.

## Immediate Implementation Plan

1. Add this ADR.
2. Add `/home/agent/bin/project-state-refresh`.
3. Generate first `latest.json` and `latest.md` for `core`.
4. Use those files as the compact resume point for future work.
5. Only after that move into the next macro-architecture slice.

## Current Baseline to Capture in the First Snapshot

The first generated state snapshot should capture at least:

- repo head: `d2dade3`
- branch: `main`
- closed slice: runtime governance / manifest / registry / docs alignment
- current state: registry-aware runtime governance flow is green
- current priority: bootstrap the project-state layer
- next macro-slice after that: integration blueprint for runtime + memory + search + Obsidian + OpenClaw

---

## Why This Comes Before Integration Blueprint

The integration blueprint is important, but it is more valuable once the project has a stable compact state layer.

Without that, each future macro-slice increases the amount of repeated orientation work.

With the state layer in place, future slices become easier to resume, verify, and transfer.

So the sequence is:

1. project-state layer
2. first project-state snapshot
3. integration blueprint
4. deeper cross-layer implementation

---

## Consequences

### Positive consequences

- less repeated manual checkpoint writing
- faster resume after pauses
- easier move to a new chat branch
- clearer distinction between active state and old artifacts
- better operational orientation before new architecture work

### Trade-offs

- one more maintained layer exists
- the refresh helper becomes part of project discipline
- stale state is possible if refresh is skipped

That trade-off is acceptable because the layer is compact and explicit.

---

## Rollback

If this layer becomes noise, rollback is simple:

- stop using `project-state-refresh`
- remove `core/artifacts/project-state/`
- keep this ADR as decision history

The rollback cost is low.

---

## Decision Summary

The `core` topic will gain a compact project-state layer.

This layer will be:

- explicit
- refreshable
- compact
- safe
- non-secret
- subordinate to real source-of-truth files

The next implementation step after this ADR is:

- create `/home/agent/bin/project-state-refresh`
- generate the first `latest.json` and `latest.md`
