# Policy — project-research

## Main principle
Collect the minimum amount of research needed for a strong start.
Do not maximize source count. Maximize relevance.

## Workflow
1. Check local memory first.
2. Decide whether external research is needed at all.
3. Choose `none`, `quick`, or `deep`.
4. Use metadata pass before detailed reading.
5. Extract only practical, planning-relevant findings.
6. Write `RESEARCH.md` and `sources.json`.
7. Hand off to `PROJECT_PLAN.md`.

## When to use `none`
- local follow-up work
- tiny changes
- strong local context already exists

## When to use `quick`
- new project
- new external dependency or platform
- useful to sample a few comparable solutions
- moderate uncertainty

## When to use `deep`
- quick mode did not produce enough signal
- official sources conflict
- multiple architecture paths exist
- technical or business cost of error is high

## Bounded source buckets
1. Similar solutions
2. Official / authoritative sources
3. Recurring risks / issue patterns
4. Practical conclusions

## Anti-bloat rules
Never:
- collect large link dumps;
- summarize full READMEs or full articles;
- scan every issue thread;
- widen scope without a reason;
- turn research into implementation.

## Output quality bar
`RESEARCH.md` must be concise, structured, and immediately useful for planning.
